# Implementation Plan: Sol API — Hono Client Data Service

**Branch**: `feature/001-hono-client-api` | **Date**: 2026-04-22 | **Spec**: [spec.md](spec.md)

## Summary

Build a Hono API deployed as a Cloudflare Worker that serves client data from the existing Neon PostgreSQL database. The notification service and future consumers (including MCP agents) will call this API instead of holding direct database connections. The API exposes five endpoints (health + 4 client CRUD routes) behind a shared API key, and strips sensitive credential fields from all responses.

---

## Technical Context

**Language/Version**: TypeScript 5.x / Node.js 20+ (Cloudflare Workers V8 runtime)
**Primary Dependencies**: `hono ^4.x`, `@neondatabase/serverless ^1.x`, `zod ^3.x`, `wrangler ^3.x` (devDep)
**Storage**: Neon PostgreSQL — existing `clients` table; no schema changes
**Testing**: Vitest + `@cloudflare/vitest-pool-workers`
**Target Platform**: Cloudflare Workers (V8 isolate runtime)
**Project Type**: Web service (HTTP API)
**Performance Goals**: p95 response time < 200ms; ~0ms cold start (V8 isolate guarantee)
**Constraints**: CF Workers free tier: 10ms CPU/request; paid ($5/mo): 50ms. Simple DB lookups fit comfortably.
**Scale/Scope**: ~10–50 clients; low-frequency reads (scheduler fan-outs + per-request lookups)

---

## Constitution Check

This is a new project — the notification service constitution does not govern `sol-api`. The relevant principles carried forward as project conventions are:

| Principle | Application in sol-api |
|-----------|----------------------|
| Multi-environment safety | `ENVIRONMENT` binding controls behavior; separate `DATABASE_URL` per environment |
| Credentials in env vars only | `DATABASE_URL` and `API_KEY` stored as Cloudflare secrets; never in `wrangler.toml` |
| Sensitive fields never exposed | `google_service_account_key` stripped from all responses at the query layer |
| Consistent error shape | All 4xx/5xx use `{ success, error: { code, message, details } }` envelope |

No violations. No complexity justification required.

## Implementation Phases

### Phase 1: Project Bootstrap

Set up the Cloudflare Workers project with Hono, TypeScript, Wrangler, and Vitest. No application logic yet — just a working skeleton that deploys and returns `GET /health`.

**Deliverables**:
- `package.json` with all dependencies
- `wrangler.toml` configured for the `sol-api` worker
- `tsconfig.json` targeting the CF Workers runtime
- `vitest.config.ts` using `@cloudflare/vitest-pool-workers`
- `src/index.ts` with Hono app + health route
- `.gitignore` covering `.dev.vars`, `dist/`, `node_modules/`
- Health check test passing

**Done when**: `npm run dev` starts, `curl localhost:8787/health` returns `{ success: true, data: { status: "ok" } }`

---

### Phase 2: Auth Middleware + Error Handler

Implement the `X-API-Key` middleware and global error handler before any data routes are added.

**Deliverables**:
- `src/middleware/auth.ts` — rejects requests without valid key with 401
- `src/middleware/error.ts` — catches `HTTPException` + unexpected errors, returns envelope shape
- `src/types/index.ts` — `Env` interface (CF Workers bindings), `ErrorCode` enum, `ApiResponse<T>` type
- Unit tests for auth middleware (valid key, missing key, wrong key)

**Done when**: Requests to any route without `X-API-Key` return `{ success: false, error: { code: "UNAUTHORIZED" } }`

---

### Phase 3: Database Layer

Wire up Neon using the fetch transport pattern appropriate for CF Workers.

**Deliverables**:
- `src/lib/db.ts` — exports `createDb(url: string)` factory that returns a `neon` tagged-template client
- `src/types/index.ts` — `ClientRow` type (matches DB columns; excludes `google_service_account_key` from response type)
- Health route updated to test DB connectivity (runs `SELECT 1`)
- Integration test: health returns `database: "connected"` with valid URL; `database: "error"` with bad URL

**Done when**: Health endpoint reflects live DB status

---

### Phase 4: Client Read Routes

Implement `GET /v1/clients` and `GET /v1/clients/:id`.

**Deliverables**:
- `src/routes/clients.ts` — list + get-by-id handlers
- List: filters `active = TRUE`; supports `limit` query param
- Get-by-id: returns 404 for missing or inactive clients
- `google_service_account_key` stripped at query level (not selected in SQL)
- Integration tests for: happy path list, empty list (200 + []), get existing, get missing (404), get inactive (404), missing auth (401), limit param

**Done when**: All read route integration tests pass

---

### Phase 5: Client Write Routes

Implement `POST /v1/clients` and `PATCH /v1/clients/:id`.

**Deliverables**:
- `src/validators/client.ts` — Zod schemas for create payload (required: id, name, email) and update payload (all optional; timezone enum validation)
- `POST /v1/clients` handler: validates → inserts → returns 201 with new record; 409 on duplicate ID; 422 on validation failure
- `PATCH /v1/clients/:id` handler: validates → updates only provided fields → returns 200; 404 if not found; 422 on bad values
- Integration tests for: create valid, create duplicate (409), create missing fields (422), create invalid timezone (422), update email, update timezone, update invalid timezone (422), update non-existent (404)

**Done when**: All write route integration tests pass

---

### Phase 6: Hardening + Deployment

Final polish before the notification service migrates to consume this API.

**Deliverables**:
- `wrangler.toml` preview and production environment sections
- Secrets documented in `quickstart.md` (`wrangler secret put` commands)
- `README.md` with project overview, commands, and endpoint summary
- CI: GitHub Actions workflow — `npm test` on push to `main` and PRs
- Deployed to Cloudflare Workers (staging URL shared with notification service team)

**Done when**: API deployed and `GET /v1/clients` returns real client data from production Neon branch
